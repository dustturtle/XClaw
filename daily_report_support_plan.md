# XClaw Daily Report Support Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add `daily_stock_analysis`-style self-selected stock daily reports and strategy-based trade points to XClaw without changing data sources or the overall agent framework.

**Architecture:** Keep the existing `agent_loop`, tools, scheduler, and multi-channel delivery intact. Add a lightweight strategy engine plus a report service that reuses current market tools, stores structured results, and exposes them through chat, scheduled jobs, and a unified admin page.

**Tech Stack:** Python 3.11+, FastAPI, SQLite via `aiosqlite`, APScheduler, existing XClaw tool registry and channel adapters.

---

## Summary

- The core value to borrow from `daily_stock_analysis` is not "more data sources", but "compress messy market data into a fixed, actionable decision dashboard".
- XClaw already has most of the foundation: `watchlist`, `stock_quote`, `stock_history`, `stock_indicators`, `stock_news`, `stock_fundamentals`, `stock_gap_analysis`, `stock_backtest`, `schedule_task`, and channel push adapters.
- The missing piece is a productized orchestration layer:
  - strategy-based trade point generation
  - watchlist daily report generation
  - report history and manual run entrypoints
  - lightweight admin UI for the investment workflow
- Scope for v1:
  - A-share daily report loop done well
  - 11 strategy names supported, with layered implementation depth
  - trade points treated as "reference trade points", never as guaranteed investment advice

## Product Behavior

### Daily report behavior

- The default stock pool comes from the existing `watchlist` table for the current `chat_id`.
- A daily report contains:
  - market overview
  - one summary conclusion for the day
  - per-stock strategy scan results
  - high-value strategy cards only
  - risk reminders
  - next-day watch items
  - disclaimer
- Reports can be triggered from:
  - chat
  - scheduled task
  - admin UI manual run button
- Reports are persisted so users can review history instead of regenerating everything on every request.

### Trade point behavior

- Trade points are generated by a new strategy layer, not invented directly by the LLM.
- Every strategy result uses a common schema:
  - `strategy_id`
  - `signal_status`
  - `bias_score`
  - `buy_zone`
  - `stop_loss`
  - `target_1`
  - `trigger_condition`
  - `risk_notes`
  - `why_not_trade`
- Point generation rules:
  - buy zones must come from deterministic anchors like MA lines, box support, breakout levels, prior structure, or strategy conditions
  - stop losses must come from deterministic invalidation points
  - targets must come from nearby resistance, box top, measured breakout extension, or fixed risk/reward templates
  - if no clean edge exists, the output must explicitly say "no trade"

### Strategy presentation behavior

- All 11 `daily_stock_analysis` strategy names are supported.
- Strategies are split into two implementation tiers.
- Rule-based strategies return direct reference trade points:
  - `bull_trend`
  - `ma_golden_cross`
  - `shrink_pullback`
  - `volume_breakout`
  - `bottom_volume`
  - `one_yang_three_yin`
  - `box_oscillation`
- Framework strategies return structured conditional trade plans instead of pretending to be fully deterministic:
  - `chan_theory`
  - `wave_theory`
  - `emotion_cycle`
  - `dragon_head`
- The system scans all strategies, but reports only show "valuable strategies":
  - triggered
  - near-trigger
  - clear risk-off / reduce-position signal
- All other strategies are collapsed into a short "no effective signal" summary.

## Implementation Changes

### 1. Add strategy engine

- Create a lightweight `TradePointEngine` under a focused module such as:
  - `xclaw/investment/strategy_engine.py`
  - `xclaw/investment/strategy_models.py`
  - `xclaw/investment/strategy_rules.py`
- This engine should:
  - fetch reusable evidence once per symbol
  - normalize it into a small internal context object
  - evaluate all strategies over that context
  - return structured outputs without channel-specific formatting
- Reused evidence sources:
  - `stock_quote`
  - `stock_history`
  - `stock_indicators`
  - `stock_gap_analysis`
  - `stock_fundamentals`
  - `stock_news`
  - `market_overview` when building the report, not per-stock
- Introduce a configurable bias discipline similar to `daily_stock_analysis`:
  - default `strategy_bias_threshold = 5.0`
  - used to prevent obvious chasing after a sharp move away from MA5

### 2. Add report service

- Create `InvestmentReportService`, for example:
  - `xclaw/investment/report_service.py`
- Responsibilities:
  - load watchlist for the active `chat_id`
  - call `TradePointEngine` for each stock
  - build a compact, fixed-shape markdown report
  - optionally call the current LLM only for summarization and wording polish, not numeric point generation
  - save report artifacts and summary metadata

### 3. Add new tools

- Add `strategy_scan` tool:
  - single stock
  - supports `strategies=all` or selected list
  - supports `output_mode=summary|full`
- Add `investment_report` tool:
  - `action=generate|latest|history`
  - defaults to using current watchlist
  - supports symbol override for ad hoc generation
- Agent routing rule additions in `xclaw/agent_engine.py`:
  - if the user asks for "买点/卖点/策略分析/决策卡", prefer `strategy_scan`
  - if the user asks for "自选股日报/盘后简报/每日决策仪表盘", prefer `investment_report`
  - keep `sub_agent` out of core trade-point calculation

### 4. Add storage

- Extend `xclaw/db.py` with new tables:
  - `investment_reports`
  - `strategy_runs`
- `investment_reports` should store:
  - `id`
  - `chat_id`
  - `report_type`
  - `title`
  - `summary`
  - `content_markdown`
  - `symbol_count`
  - `trigger_source`
  - `created_at`
- `strategy_runs` should store:
  - `id`
  - `chat_id`
  - `symbol`
  - `market`
  - `strategies_json`
  - `valuable_strategies_json`
  - `created_at`
- Keep isolation aligned with existing `chat_id` scoping.

### 5. Reuse scheduler instead of creating a new job system

- Keep `scheduled_tasks` as the only scheduling mechanism.
- Daily report scheduled jobs should just point to a fixed action prompt or direct report action.
- No second task table, no second scheduler abstraction.

### 6. Add unified admin entry

- Extend `xclaw/channels/web.py` with a new admin page, preferably `/admin`.
- Keep implementation lightweight:
  - inline HTML
  - inline JS
  - no new frontend build system
- The admin page should aggregate:
  - daily report list
  - report detail drawer
  - manual run
  - watchlist management
  - strategy scan detail view
  - scheduled report tasks
  - existing session/config/channel status sections where useful
- "Unified admin" in this plan means:
  - one entry page for the investment reporting workflow
  - reuse current XClaw APIs and status views where possible
  - not a full rewrite of the entire XClaw platform UI stack

## Acceptance Criteria

- A user can ask in chat for a watchlist daily report and receive a persisted markdown report.
- A user can ask in chat for strategy-based buy/sell analysis on a single stock and receive strategy cards with reference trade points.
- Scheduled tasks can generate and deliver the report using existing channel push paths.
- The admin page can:
  - manually run a report
  - list report history
  - inspect report details
  - manage watchlist items
  - manage report tasks
- Strategy cards never output unsupported fabricated numeric levels when the evidence is insufficient.
- A-share v1 behavior is stable even when some news/fundamental data calls fail.

## Assumptions

- First version promises full daily report quality only for A-shares.
- "Reference `daily_stock_analysis` strategies" means reusing its strategy naming, framing, discipline, and output style, not copying internal implementation one-to-one.
- "All strategies supported" means all 11 strategies are exposed, but only 7 are implemented as hard rules in v1.
- Trade points are always advisory and must include invalidation logic.
- Watchlist remains the default stock source; no separate report-only stock grouping model is added in v1.
- The admin page is unified for this workflow, but not a full platform rewrite.
